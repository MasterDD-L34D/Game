# V3 canonical vs V3.2 framing — Reconciliation analysis

**Trigger**: design pitch v3.2 ricevuto da designer esterno. Diverge in punti rilevanti dal v3 canonical chiuso 2026-04-26 (post 6 design call + master-dd survey 15 risposte).

**Scopo**: matrix completa allineamento/conflict + adoption recommendation per ogni v3.2 decision + risposta alle 3 micro-Q residue.

**Authority**: A4 (proposal report). Non sostituisce v3 canonical; informa la decision master-dd su quali pezzi v3.2 adottare/rigettare/ibridare.

## 1. TL;DR (5 bullet)

1. **Allineamento ~55%, conflict ~30%, gap ~15%.** v3.2 e v3 condividono il quadro generale (compromessi accettati: failure as lore, no game-over manuale, no map travel, lore emergente, 4 schermate vertical slice). Divergono su 3 dimensioni identitarie strutturali (A.L.I.E.N.A. / VC axes / Forma+Stadio+Archetipi+MBTI stack).
2. **Top 3 conflict bloccanti**: (a) **A.L.I.E.N.A.** v3 = acronimo metodo design 6-dim ✱ v3.2 = AI entity con voce — confitto ontologico; (b) **VC axes**: 6-axis canonical (`aggro/risk/cohesion/setup/explore/tilt`) live in `data/core/telemetry.yaml:27-33` + engine `apps/backend/services/vcScoring.js:38-90` ✱ v3 propone 5 axes UI surface diversi creature-themed ✱ v3.2 prende le 6 axes giuste ma fonde col 5-swipe-input ✱ engine MBTI 4-axes a parte; (c) **identity stack**: v3 ha 16 MBTI Forms + Stadio I-X anatomico + Lineage emergente ✱ v3.2 aggiunge 3 archetipi cross-specie (Agile/Robusta/Specializzata) come quarta dimensione.
3. **Adopt-rate raccomandato**: 12 v3.2 decisions → **5 ADOPT** (timing 75s, no tutorial, atollo soft, Atollo+Frattura pitch, lore emergente, 1-file HTML), **3 HYBRID** (Stadio I-X, 4 schermate, mating death-trigger), **3 OVERRIDE** (A.L.I.E.N.A. = acronimo, Skiv canonical voce, 6-axis VC vs 5-swipe), **1 REJECT** (3 archetipi cross-specie redundant).
4. **Q1/Q2/Q3 raccomandazioni coerenti**: Flint name reveal **mai** (Skiv canonical sostituisce); A.L.I.E.N.A. first words **mai-direct** (è acronimo metodo, non parla — Skiv parla per lei); mockup language **IT puro narrative + EN code/UI label** (style guide `00E-NAMING_STYLEGUIDE.md` già canonical).
5. **Migration effort**: ~6-9h se HYBRID accept (rinomina solo "Flint" → Skiv-instance nelle 4 schermate vertical slice + mantieni 75s/atollo/4-schermo/lore-emergente come è); ~25-35h se ADOPT integrale (refactor identity stack 4-dim + UI 5-axes + nuova surface). **Verdict pragmatico: HYBRID**.

### 3.3 Identity stack — 4 dimensioni (Forma / Stadio / Archetipi / MBTI) sostitutive o ortogonali?

**Problema**: v3.2 introduce **3 archetipi cross-specie** (Agile / Robusta / Specializzata) come dimensione di "chi sei". v3 ha già 3 dimensioni canonical (Stadio I-X anatomico + 16 MBTI temperamento + Lineage tribe emergente).

**Stato canonical**:

- `data/core/forms/mbti_forms.yaml:11-60`: 16 Forms (Custode/Stratega/Mediatore/Analista/etc.)
- `data/core/species/dune_stalker_lifecycle.yaml`: 5 macro phase + sub-divisione 10 Stadi I-X (`docs/planning/2026-04-27-forme-10-stadi-naming-spec.md`)
- `feedback_tribe_lineage_emergent_breakthrough.md`: Tribe = lineage emergent post-mating (Sprint D shipped #1877)
- `data/core/jobs.yaml` + `jobs_expansion.yaml`: 11 jobs runtime (vanguard/skirmisher/.../stalker/symbiont/etc.) **= specializzazione tattica**

**Domanda chiave**: i 3 archetipi v3.2 sono ridondanti (collision con jobs?) o complementari (nuova dimensione)?

**Analysis**:

- **Agile** ≈ skirmisher / stalker (DEX-focused jobs già in `jobs.yaml`)
- **Robusta** ≈ vanguard / warden (HP/DEF-focused jobs)
- **Specializzata** ≈ artificer / invoker / harvester (utility/ranged/eco jobs)

→ **3 archetipi v3.2 = abstraction sopra i 7+4 jobs canonical**, niente di nuovo. Player-felt simplification (3 invece di 11), ma scarda mapping job→archetipo runtime.

**Opzioni reconciliation**:

| Opzione | Approccio                                                                                 | Effort | Pro                                  | Contro                                       |
| ------- | ----------------------------------------------------------------------------------------- | -----: | ------------------------------------ | -------------------------------------------- |
| **A**   | **REJECT 3 archetipi v3.2** — il job system canonical fa già il loro lavoro              | ~0h    | Zero migration                       | Designer esterno deve accettare              |
| B       | HYBRID — 3 archetipi = label UX pulse-3-shop pre-Nexus, mapping a job concrete dopo     | ~4h    | Player-felt onboarding semplificato | Aggiunge layer + naming collision con jobs   |
| C       | ADOPT — sostituisci 11 jobs con 3 archetipi in surface UI                                | ~20h   | Massima semplicità player           | Distrugge granularità tactical jobs canonical |

**Raccomandazione**: **Opzione A — REJECT 3 archetipi cross-specie**.

Razionale:
1. **Tribe = lineage emergent** già copre "chi sei culturalmente" (post-mating)
2. **Job runtime** copre "cosa fai tatticamente" (vanguard/skirmisher/etc., già 11 entries)
3. **Stadio I-X** copre "quanto sei sviluppato"
4. **16 MBTI Forms** copre "che temperamento hai"
5. → 4 dimensioni canonical sufficienti. **Aggiungere 3 archetipi v3.2 = sovrapposizione jobs**, dilatazione design senza valore aggiunto.

Counter-proposal: se v3.2 vuole "3 categorie player-felt", spec fa **collassare i 16 MBTI Forms in 3 cluster temperamentali** (es. Aggressive/Cautious/Adaptive) come UI macro-pre-pick, NON aggiungere dimensione.

### Q2. A.L.I.E.N.A. first words — quando?
